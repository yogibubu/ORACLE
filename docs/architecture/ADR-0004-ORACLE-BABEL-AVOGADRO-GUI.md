# ADR 0004: ORACLE Babel, Avogadro And GUI Visualization

Date: 2026-06-26

## Decision

The ORACLE preprocessing layer is a Babel-like translator. It is the only
normal entry point for external molecular sources:

- plain XYZ;
- enriched XYZ;
- QM-program inputs and outputs;
- SMILES through RDKit;
- future Z-matrix and legacy formats.

Its output is always an Avogadro-compatible enriched XYZ file. The file starts
with a normal XYZ block so Avogadro can view and edit the molecular geometry.
ORACLE metadata are stored after the first XYZ frame in named sections.

## Pipeline

```text
external source
  -> oracle babel/import adapter
  -> enriched XYZ first block
  -> #SOURCE
  -> #BASIC
  -> #SYMMETRY
  -> #TOPOLOGY
  -> #SYNTHONS
```

The initial structural analysis is done once:

1. normalize/import geometry;
2. determine symmetry using explicit thresholds;
3. build topology once: bonds, rings, aromaticity and descriptor graph;
4. compute and save atomic synthon descriptors;
5. let downstream tools reuse these sections.

GICForge then consumes the saved geometry/topology/symmetry state to determine
GICs and symmetrize them when requested. MORPHEUS, GF/PED, DVR, VPT2/VCI and
GUI workflows must not redo import parsing or private topology perception.

## Avogadro Contract

The enriched XYZ must remain a valid ordinary XYZ file at the top:

```text
N
comment
Atom x y z
...

#SOURCE
...
```

Avogadro view/edit works on the first XYZ block. If the user edits geometry in
Avogadro and imports it back, ORACLE replaces only the first XYZ block and
preserves metadata sections. If geometry changes, cached sections that depend on
geometry must be marked stale or regenerated by the Babel/preprocessing step.

## GUI Contract

The GUI uses the enriched XYZ as its state source:

- show the current molecule from the enriched XYZ;
- open Avogadro/Avogadro2 in view or edit mode;
- import edited coordinates by replacing the first XYZ block;
- show a 2D depiction for task navigation.

The 2D depiction follows the current Merlino rule: prefer `#SMILES` and RDKit
2D depiction when available. If `#SMILES` is absent, the GUI may later fall back
to a topology-derived depiction, but that fallback must use the saved
`#TOPOLOGY`/`#SYNTHONS` sections rather than reparsing chemistry privately.

## Consequences

- `oracle-babel` is the only place that turns external formats into ORACLE
  state.
- The enriched XYZ is both human-editable with Avogadro and machine-readable by
  ORACLE tools.
- GUI rendering is a view of the file state, not a parallel in-memory model.
- Editing geometry invalidates or regenerates dependent sections.
